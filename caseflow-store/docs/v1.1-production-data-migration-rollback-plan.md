# CaseFlow Books v1.1 Production Data Migration And Rollback Plan

- Status: Accepted for D23-T02 planning
- Date: 2026-07-16
- Related roadmap task: `D23-T02 - Plan Production Data Migration And Rollback`
- Related migration draft: `caseflow-store/supabase/migrations/0006_caseflow_books_schema_draft.sql`
- Stable fallback release: `v1.0.0`

## Purpose

This plan defines how CaseFlow Store `v1.0.0` production data is protected
before the CaseFlow Books `v1.1` schema is applied, how the schema should be
introduced, and how to return to the last stable production release if the
migration fails.

D23-T02 is documentation only. It must not apply SQL, run destructive database
operations, seed book data, create users, deploy code, or change production
environment variables.

## Migration Strategy

Use an expand-and-contract migration.

In D23, the database expands to support CaseFlow Books while preserving the
released phone-accessory store. The `categories`, `products`, existing
`profiles`, existing `orders`, and existing `order_items` remain in place.
No phone-accessory data is converted into book data because the domains are not
equivalent.

The CaseFlow Books catalog starts as new v1.1 data in D24. The old
phone-accessory catalog becomes legacy production data that can be hidden by
v1.1 repositories and UI, but it must not be deleted during D23.

## Data Handling Decision

| Data area | D23 decision | Reason |
|---|---|---|
| `categories` | Preserve in place | Keeps v1.0.0 catalog compatible during rollback |
| `products` | Preserve in place | Phone products are not valid book editions |
| `orders` | Preserve in place and expand columns | Historical orders must remain auditable |
| `order_items` | Preserve legacy product snapshots and add nullable book snapshot columns | Supports both v1.0.0 and v1.1 during transition |
| `profiles` | Preserve users and add optional v1.1 fields | Existing admin/customer identities remain valid |
| Book catalog tables | Create empty schema first, seed in D24 | Separates schema risk from data/content risk |
| Book covers/descriptions | Do not import in D23 | Content policy and safe assets are handled in D24 |

D23-T03 must create at least one production backup/export artifact before

Required evidence:

- Supabase dashboard backup, point-in-time recovery restore point, or equivalent
  provider backup evidence.
- A schema export saved under `caseflow-store/.agent/artifacts/d23-t03-backup/`.
- A data export saved under `caseflow-store/.agent/artifacts/d23-t03-backup/`.
- A short manifest recording timestamp, environment, project reference, export
  filenames, and verification notes.

Recommended artifact names:

```text
caseflow-store/.agent/artifacts/d23-t03-backup/schema-before-v1-1.sql
```

## Pre-Migration Checks

Capture these checks before applying SQL:

```sql
select count(*) as category_count from public.categories;
select count(*) as product_count from public.products;
select count(*) as order_count from public.orders;
select count(*) as order_item_count from public.order_items;
select count(*) as profile_count from public.profiles;
select role, count(*) from public.profiles group by role order by role;
select count(*) as direct_book_orders
from public.order_items
where book_edition_id is not null or product_id is null;
```

Expected state before D23-T03:

- `categories` and `products` contain the released phone-accessory catalog.
- `orders` and `order_items` may contain production or QA history and must not
  be deleted.
- `profiles.role` contains only roles accepted by v1.0.0, usually `admin` and
  `customer`.
- `direct_book_orders` is `0` because v1.1 book checkout is not live yet.

Also run the smallest meaningful application checks before migration:

```bash
npm run lint
npm run build
```

Recommended production smoke checks:

- Home page returns `200`.
- Product/category APIs return v1.0.0 catalog data.
- Anonymous admin access returns `401`.
- Existing checkout flow can create an order in a controlled QA run.
- QA order/user cleanup returns `0` remaining test artifacts.

## Migration Execution Plan

1. Confirm the backup/export evidence exists.
2. Confirm the pre-migration checks are recorded.
3. Re-inspect the SQL draft for accidental destructive operations.
4. Apply `0006_caseflow_books_schema_draft.sql` in one transaction through the
   verified Supabase apply path.
5. Do not seed book catalog data in D23-T03.
6. Do not deploy v1.1 UI/API behavior in D23-T03.
7. Record the SQL apply result, timestamp, and operator notes.

The migration draft is intended to be additive. If PostgreSQL rejects any
statement before `commit`, the transaction should fail without partial schema
application. If the apply path does not guarantee transaction behavior, D23-T03
must stop and choose a safer apply path before touching production.

## Post-Migration Database Checks

Run these after SQL application:

```sql
select to_regclass('public.book_categories') as book_categories;
select to_regclass('public.book_works') as book_works;
select to_regclass('public.book_editions') as book_editions;
select to_regclass('public.book_promotions') as book_promotions;
select to_regclass('public.book_inventory_adjustments') as book_inventory_adjustments;

select column_name
from information_schema.columns
where table_schema = 'public'
  and table_name = 'orders'
  and column_name in (
    'customer_id',
    'payment_method',
    'payment_status',
    'shipping_method',
    'tax_estimates',
    'fee_estimates',
    'display_estimate',
    'total_vnd'
  )
order by column_name;

select column_name
from information_schema.columns
where table_schema = 'public'
  and table_name = 'order_items'
  and column_name in (
    'book_edition_id',
    'book_work_id',
    'edition_title',
    'edition_language',
    'edition_format',
    'unit_price_vnd',
    'line_total_vnd'
  )
order by column_name;

select relname, relrowsecurity
from pg_class
where relnamespace = 'public'::regnamespace
  and relname in (
    'book_categories',
    'book_authors',
    'book_translators',
    'book_publishers',
    'book_cover_assets',
    'book_works',
    'book_work_authors',
    'book_work_categories',
    'book_editions',
    'book_edition_translators',
    'customer_addresses',
    'book_promotions',
    'book_inventory_adjustments',
    'orders',
    'order_items',
    'profiles'
  )
order by relname;

select proname
from pg_proc
where pronamespace = 'public'::regnamespace
  and proname = 'create_book_order_with_items';
```

Required results:

- All expected book tables resolve to non-null `to_regclass` values.
- Expected v1.1 columns exist on `orders` and `order_items`.
- RLS is enabled on all new public tables and remains enabled on existing
  protected tables.
- `create_book_order_with_items` exists.

If full Playwright is too slow for the first D23-T03 pass, run a focused

Required production checks:

- Cart validation still recalculates server-side prices.
- Existing simulated checkout still creates a v1.0.0 order.
- Admin login and order status update still work.

- No book catalog data is visible yet unless D24 seed has run.

## Rollback Decision Tree

### Case 1: SQL Fails Before Commit

Action:

- Let the transaction roll back.
- Do not retry immediately.
- Save the error output.
- Inspect the failing statement and update the migration draft in a follow-up
  task before another apply attempt.

Expected recovery:

- Production stays on `v1.0.0` database behavior.
- Run the pre-migration smoke checks again.

### Case 2: SQL Applies, But v1.0.0 App Still Works

Action:

- Keep the additive schema in place.
- Do not run destructive rollback.
- Continue to D24 only after all D23-T03 checks pass.

This is the preferred outcome. Additive schema by itself should not force an
app rollback.

### Case 3: SQL Applies, But v1.0.0 App Breaks

Action order:

1. Roll back the application deployment to `v1.0.0` or the last known stable
   Vercel production deployment.
2. Rerun storefront, checkout, admin, and RLS smoke checks.
3. If the app rollback fixes production, leave the additive DB schema in place
   and create a fix task before continuing v1.1 work.
4. If the app remains broken because of DB changes, use the D23 rollback SQL
   path or restore from backup.

### Case 4: Data Corruption Or Missing Production Data

Action:

- Stop all further migration work.
- Do not seed books or deploy v1.1 code.
- Restore from Supabase backup/export or provider point-in-time recovery.
- Verify table counts, order integrity, and admin access against the
  pre-migration evidence.

This is the only rollback case where provider backup restore should be the
default instead of additive-schema cleanup.

## D23 Rollback SQL Shape

The D23 rollback path is valid only before v1.1 book seed/order workflows have
created real data. Before running any rollback SQL, verify:

```sql
select count(*) as book_order_item_count
from public.order_items
where book_edition_id is not null or product_id is null;

select count(*) as staff_profile_count
from public.profiles
where role = 'staff';
```

Both counts must be `0` before the D23 cleanup rollback is allowed.

Rollback cleanup should remove v1.1 objects in dependency order:

1. Revoke and drop `public.create_book_order_with_items`.
2. Drop book-only policies and grants.
3. Drop `book_inventory_adjustments`.
4. Drop `book_edition_translators`.
5. Drop `customer_addresses`.
6. Drop `book_promotions`.
7. Drop `book_editions`.
8. Drop `book_work_categories`.
9. Drop `book_work_authors`.
10. Drop `book_works`.
11. Drop `book_cover_assets`.
12. Drop `book_publishers`, `book_translators`, `book_authors`, and
    `book_categories`.
13. Drop v1.1-only indexes on `orders` and `order_items`.
14. Drop v1.1-only columns and constraints from `order_items`.
15. Restore `order_items.product_id` to `not null` only if no row has null
    `product_id`.
16. Drop v1.1-only columns and constraints from `orders`.
17. Restore `profiles_role_check` to `role in ('customer', 'admin')` only if no
    profile has role `staff`.
18. Drop optional v1.1 profile columns only if no customer profile data has been
    collected.

After D24 or any live v1.1 order/customer workflow, this cleanup rollback is no
longer safe as a default. Use backup restore or a new migration-specific
rollback plan.

## Stop Conditions

Stop D23-T03 immediately if any of these happen:

- No backup/export evidence exists.
- Pre-migration production smoke fails.
- SQL apply path cannot guarantee transaction behavior.
- SQL inspection finds a destructive operation against `categories`,
  `products`, existing `orders`, existing `order_items`, or existing
  `profiles`.
- Post-migration RLS check shows direct public writes to operational tables.
- Existing v1.0.0 checkout or admin order operations fail after migration.
- Any production data count unexpectedly decreases.

D23-T03 can be accepted only if:

- The backup/export evidence exists.
- The schema apply succeeds.
- v1.0.0 production behavior still works.
- RLS deny-by-default posture is preserved.
- No legacy phone-accessory data was deleted or converted.
- Book seed data remains pending for D24.
