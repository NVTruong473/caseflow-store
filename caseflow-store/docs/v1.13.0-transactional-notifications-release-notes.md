# v1.13.0 - Transactional Notifications And Simulated Transfer Operations

Date: 2026-07-23

## Summary

This release adds a durable transactional-notification boundary to CaseFlow
Books. Customers receive account-scoped order activity in the application;
staff/admin gain minimized delivery operations; and simulated bank-transfer
decisions are protected server-side. External email/SMS delivery remains
disabled in Production until approved provider credentials exist.

## Customer Experience

- Added an account activity inbox for order creation and lifecycle updates.
- Added unread state and account-scoped mark-read behavior.
- Added verified email/phone identity status to the account surface.
- Added optional SMS OTP request/verification controls that appear only when a
  live SMS provider and hashing secret are configured.
- Preserved order history, customer cancellation, checkout, vouchers, and
  stable public tracking behavior.

## Operations

- Added the `/admin/notifications` workspace for staff/admin delivery status,
  event/channel filtering, and guarded retries.
- Admin can inspect a sanitized provider-readiness summary; staff cannot access
  provider configuration.
- Added explicit confirm/reject controls for simulated bank transfers. The
  customer cannot mark an order paid.
- Operational DTOs omit contact data, OTP material, raw event metadata,
  provider responses, and secrets.

## Architecture And Data

- Added centralized bilingual templates, provider interfaces, a dispatcher,
  retry/backoff policy, and Resend/Twilio adapters behind `disabled`, `sandbox`,
  and `live` modes.
- In-app delivery is authoritative. Sandbox external delivery records an
  honest staff-only preview and performs no network request.
- Added migrations `0012_transactional_notifications.sql` and
  `0013_notification_order_lifecycle.sql` for the outbox, customer inbox,
  hashed phone challenges, RLS/grants, dispatch claim/read/verification RPCs,
  idempotent event-channel keys, and deletion-safe notification history.
- Migration apply evidence preserved the 13 pre-existing orders, 13 order
  items, and 27 account vouchers.
- Order creation, customer cancellation, staff/admin updates, and simulated
  transfer decisions dispatch best-effort lifecycle events after committed
  business mutations.

## Security And Production Controls

- OTP values are never stored raw or returned by an API; challenges are
  customer-owned, HMAC-protected, expiring, resend-limited, and attempt-limited.
- The internal dispatcher requires a timing-safe bearer secret.
- Anonymous access to all new customer/admin/OTP/dispatcher/transfer endpoints
  returns the stable `401/UNAUTHORIZED` contract.
- Production has no notification provider or dispatcher variables configured,
  so external delivery fails closed as `disabled`.
- No real payment, bank account, SMTP, SMS sender, or payment-provider secret
  was introduced. QR simulation remains locked in Production.

## Remaining Boundaries

- Email notifications need an approved Resend key and sender domain before
  `live` mode can be enabled.
- SMS/phone verification needs Twilio credentials, an approved sender, and a
  production OTP hashing secret.
- Supabase Auth confirmation still uses its separately configured sender; this
  release does not replace that service.
- Payments remain simulated. Real settlement, reconciliation, refund, carrier,
  and warehouse integrations remain outside this release.
