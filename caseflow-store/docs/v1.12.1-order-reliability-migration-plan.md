# v1.12.1 Order Reliability Migration Plan

## Scope

Migration `0011_atomic_idempotent_book_orders.sql` is additive:

- add nullable `orders.checkout_attempt_id`;
- add a partial customer/attempt unique index;
- correct the voucher/order history foreign key to `ON DELETE RESTRICT`, which
  matches the existing used-voucher shape constraint;
- add `create_book_order_with_items_v2`;
- retain the released `create_book_order_with_items` RPC.

It does not update or delete an existing order, order item, voucher, customer,
catalog row, payment, or inventory record.

## Apply

1. Run the static contract verifier.
2. Run the database script without apply mode and record preflight counts.
3. Set `ORDER_RELIABILITY_APPLY=true` for one controlled apply.
4. Verify row counts are unchanged and both old/new RPCs exist.
5. Run the dedicated retry/voucher E2E before deploying the app.

The apply script reads `SUPABASE_DB_URL` from the process environment and does
not print it. Because the direct project host is IPv6-only from the current
network, the script prefers the Supabase session pooler and falls back to the
direct URL. The repository `.env.local` remains untracked.

## Rollback

Before application deployment, rollback can remove the new function, index,
and nullable column because the released app still uses the retained RPC:

```sql
begin;
drop function if exists public.create_book_order_with_items_v2(
  uuid, text, uuid, text, text, text, jsonb, text, text, integer, integer,
  integer, integer, jsonb, jsonb, jsonb, text, jsonb
);
drop index if exists public.orders_customer_checkout_attempt_unique_idx;
alter table public.orders drop column if exists checkout_attempt_id;
commit;
```

After application deployment, first roll the app back to `v1.12.0`. Removing
the additive database objects is optional; retaining them is safer because
they do not affect the old RPC or existing rows.
