# v1.12.1 - Atomic Order Reliability Patch

Date: 2026-07-22

## Summary

This patch closes a checkout consistency gap left by the earlier account
voucher workflow. One customer checkout attempt is now idempotent, and order,
order-item, and eligible signup-voucher writes commit in one PostgreSQL
transaction.

## Runtime Changes

- Checkout creates and reuses a session-scoped UUID until an order succeeds.
- `POST /api/orders` validates the UUID but still ignores browser totals,
  prices, stock, promotion ownership, and role claims.
- The order use case recovers an existing customer/attempt order before
  creating another one and repeats that read across the promotion race window.
- The repository uses versioned RPC `create_book_order_with_items_v2`.
- Split signup-voucher reserve/confirm writes were removed from order creation.
- Existing API envelopes, error codes, checkout copy, pricing, tax, shipping,
  payment, auth, and authorization rules remain unchanged.

## Database Changes

Migration `0011_atomic_idempotent_book_orders.sql`:

- adds nullable `orders.checkout_attempt_id` for backward compatibility;
- adds a partial unique index on customer plus attempt ID;
- adds the versioned atomic/idempotent order RPC;
- retains the released order RPC for rollout compatibility;
- aligns used-voucher order history with `ON DELETE RESTRICT`, matching the
  existing constraint that a used voucher must retain its order reference.

The migration is additive for order data. Apply verification recorded unchanged
order, item, and voucher counts across the migration transaction.

## Regression Coverage

- Two concurrent requests with one attempt ID return the same order.
- A later retry returns the same order.
- Only one order row exists for the attempt.
- The signup voucher is linked exactly once.
- A different attempt cannot reuse the consumed voucher.
- Signup voucher and QR verifier cleanup now removes test voucher rows before
  test orders so evidence runs do not leave data behind.
