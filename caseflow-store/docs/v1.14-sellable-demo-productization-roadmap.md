# v1.14 Sellable Demo Productization Roadmap

Date: 2026-07-24

## Objective

Turn CaseFlow Books into a more credible sellable source-code demonstration
without pretending that the reference deployment is an operating legal
business. The phase improves configuration, buyer discovery, handoff,
verification, and release consistency while retaining the single-store Next.js
modular monolith.

## Non-Goals

- Multi-tenant SaaS or reseller platform.
- No-code theme/catalog builder.
- Real payment, bank, email, SMS, carrier, warehouse, accounting, or tax
  integration.
- Buyer-specific legal advice or fabricated corporate/contact details.
- Automatic migration from an unknown buyer catalog.

## Tasks

### PRODUCTIZE-T01 - Accept ADR And Audit Baseline

Acceptance criteria:

- ADR-0017 is accepted and indexed.
- The audit identifies concrete buyer-readiness gaps with file evidence.
- Adversarial questions distinguish a product demo from a turnkey business.
- Scope explicitly excludes real integrations and invented business identity.

Verification:

- ADR/index links resolve.
- `git diff --check` passes.

### PRODUCTIZE-T02 - Centralize Public Store Configuration

Acceptance criteria:

- Store name, short mark, bilingual tagline, support hours, optional phone/
  email, legal display name/year, and canonical URL have typed defaults.
- Header, footer, root metadata, SEO helpers, robots/sitemap, and payment
  merchant fallback use the shared values where appropriate.
- Unconfigured support contacts are omitted rather than fabricated.
- No server secret enters the client bundle.

Verification:

- TypeScript, ESLint, build, configuration verifier, and focused browser checks
  pass.

### PRODUCTIZE-T03 - Create Buyer Customization And Data Handoff

Acceptance criteria:

- A buyer discovery questionnaire records brand, legal, catalog, payment,
  fulfillment, roles, migration, and acceptance decisions.
- A customization guide maps environment values and source/data ownership.
- Catalog replacement documents validation, order snapshots, rollback, cover
  provenance, and production migration gates.
- No guide claims that unknown buyer data can be imported without mapping.

Verification:

- Link and command examples are checked against the repository.
- Required/optional/blocked values are unambiguous.

### PRODUCTIZE-T04 - Create Demo, Operations, And Commercial Handoff Pack

Acceptance criteria:

- A short demo script covers customer, staff, and admin value without real
  provider claims.
- A runbook covers deployment, rollback, backups, incidents, test-data cleanup,
  credential rotation, and provider activation gates.
- A commercial-boundary checklist distinguishes included source, buyer work,
  third-party costs, and acceptance evidence.
- README and release references are current.

Verification:

- All internal links resolve.
- No placeholder hotline/mailbox or stale current-release claim remains.

### PRODUCTIZE-T05 - Add Productization Verifier And Fix Findings

Acceptance criteria:

- An automated verifier checks central configuration usage, stale release
  references, placeholder contacts, unsafe business claims, and required
  handoff documents.
- The verifier is available as an npm script and emits JSON evidence.
- All high-confidence findings are fixed without broad feature expansion.

Verification:

- Productization verifier passes with zero findings.
- Architecture, notification, payment-safety, secret, and cleanup gates remain
  green.

### PRODUCTIZE-T06 - Full QA And Release

Acceptance criteria:

- Lint, TypeScript, build, local/Production browser tests, responsive and
  accessibility checks pass.
- Production keeps real providers disabled and mock payment locked.
- Deployment, tag, release notes, and consistency audit identify the reference
  app as a sellable demo, not an operating business.
- Worktree and remote state are clean and verified.

Verification:

- Every gate is reported as PASS, FAIL, BLOCKED, or NOT RUN.
- Release occurs only after all release-blocking gates pass.
