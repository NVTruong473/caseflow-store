# CaseFlow Books v1.2 Provenance And Content Quality Contracts

Status: accepted for `V12-T04` on 2026-07-17.

## Decision

CaseFlow Books uses a new catalog-specific `CatalogProvenanceRecord` rather
than expanding the legacy `SourceNote`. `SourceNote` remains the stable
v1.1 contract for tax, fee, currency, author, and placeholder-source notes.
Changing it would create an unrelated breaking change.

The v1.2 contract is implemented in:

- `src/types/content-provenance.ts`;
- `src/lib/validation/content-provenance.ts`;
- `src/lib/content/content-quality.ts`;
- `src/lib/api/content-provenance.ts`.

## Provenance Record

Every record identifies:

- target entity, entity ID, and field key;
- source label, stable source URL when applicable, and checked date;
- content kind: bibliographic fact, project-written text, or media;
- rights basis: factual data, project-created, generated original, licensed, or
  public domain;
- rights-basis note, license data, and attribution requirements;
- review status, internal reviewer note, and review date;
- source-edition key and edition-match confidence when the record describes an
  edition-specific bibliographic fact.

The contract deliberately separates content kind from rights basis. A media
record cannot claim `factual-data`, project-written text must be
`project-created`, and a bibliographic fact must be `factual-data`.

## Rights Validation

| Record | Required rules |
|---|---|
| Internal media | `media` + `project-created`; no external license record |
| Generated media | `media` + `generated-original`; no external license record |
| Licensed media | stable source URL, license name/URL/terms, explicit attribution object |
| Public-domain media | stable source URL, public-domain marker/license record, rights note, explicit attribution object |
| Bibliographic fact | stable source URL; `factual-data`; edition key and match confidence when edition-scoped |
| Project-written text | `project-written-text` + `project-created`; no external license record |

When attribution is required, both display text and display location are
mandatory. Approved or rejected records require `reviewedAt`; draft and
needs-review records cannot claim a completed review.

## Edition Integrity

`EditionProvenanceSet` covers publisher, ISBN, publication year, page count,
dimensions, weight, format, title/subtitle, and translator facts.

Every primary fact in one set must:

- target the same CaseFlow edition;
- use matching `field` and provenance `fieldKey`;
- be a bibliographic fact;
- have only one primary record per field;
- resolve to the same stable `sourceEditionKey`.

A set containing facts from different source-edition keys is rejected. This
prevents an ISBN from one publication, a page count from another, and a
publisher from a third from being silently presented as one sellable edition.

## Content Quality

The checklist separates release blockers from optional enrichment.

Blocking requirements cover title, authors, language, format, price, stock,
category, bilingual summary, primary cover, source review, edition-fact
consistency, and rights completeness.

Optional requirements cover ISBN, publisher, translator, page count,
publication date, dimensions, weight, and table of contents. An optional value
may remain absent, but any value displayed as a fact must still be verified.

Evidence states are:

- `verified`: requires a provenance record ID and receives credit;
- `missing`: receives no credit;
- `unverified`: requires an explanation and receives no credit;
- `not-applicable`: requires an explanation, receives no credit, and is
  permitted only for optional requirements.

`releaseReady` becomes true only when every blocking requirement is verified.
The quality score counts verified applicable checks only; raw field presence,
candidate values, and unreviewed data do not improve it.

## Public API Boundary

`toPublicCatalogProvenance` returns only approved records and allowlists safe
fields: source label/URL, checked date, content kind, rights basis, license,
attribution, and the literal approved status.

It intentionally omits internal reviewer notes, rights-analysis notes,
source-edition matching keys, entity IDs, field keys, and review timestamps.
The existing public book catalog serializer also continues to select explicit
public fields, so attached internal provenance objects cannot leak by object
spread.

## Database Decision

An additive database migration is required because v1.1 cannot persist
provenance for works, editions, or publishers and its cover `source_note`
cannot represent these contracts.

No SQL migration is created in `V12-T04`. The exact table, foreign-key, index,
and RLS shape must wait until the canonical catalog, cover pipeline, bilingual
content, and merchandising requirements are frozen in `V12-T05` through
`V12-T09`. The reversible migration belongs to `V12-T10`.

No existing columns, v1.1 data, IDs, slugs, API envelopes, or runtime behavior
are changed by this task.
