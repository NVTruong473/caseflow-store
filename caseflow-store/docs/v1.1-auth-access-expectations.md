# CaseFlow Books Auth Access Expectations

Status: active from `D31-T01`

## Scope

This note documents the access expectations introduced with the customer account
pages. It does not grant checkout completion, profile/address completion, order
history, or staff operations by itself; those remain later roadmap tasks.

## Actors

| Actor | Expected access after D31-T01 |
|---|---|
| Anonymous visitor | Browse storefront, open cart, visit `/account`, sign up, sign in. |
| Customer | Visit `/account`, see account state, sign out, continue to later checkout flows when implemented. |
| Staff/operator | Can hold a Supabase session and see account state, but customer account UI does not grant admin-only permissions. Staff operation screens are introduced in Day 35. |
| Admin | Can hold a Supabase session, see account state, and use the separate admin workspace. Admin authorization still depends on server-side `profiles.role` checks. |

## Verification Boundaries

- Customer sign-up and sign-in use Supabase Auth.
- Customer profiles are created or backfilled server-side; browser payloads
  cannot set `role`.
- Email confirmation is described as depending on Supabase Auth configuration.
- The UI does not claim phone verification.
- Logout clears the current Supabase session cookie.
- Admin and customer auth share the Supabase session cookie, but admin access
  remains protected by server/API role checks.
