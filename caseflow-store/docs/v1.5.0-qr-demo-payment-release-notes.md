## What Changed

- Added a server-owned payment boundary for QR payment sessions, with a shared
  provider interface for `MOCK_GATEWAY` and `DEMO_VIETQR`.
- Added a `payments` table and idempotent `mark_demo_payment_paid` RPC so order
  status and payment status are stored separately and updated consistently.
- Added authenticated customer payment APIs:
  `POST /api/payments`, `GET /api/payments/[paymentId]`,
  `POST /api/dev/payments/[paymentId]/simulate-success`, and
  `POST /api/webhooks/mock-payment`.
- Added a QR payment waiting page at `/checkout/payment` with QR rendering,
  server-time countdown, polling, loading/error states, paid-state refresh, and
  mobile/desktop layout coverage.
- Added VietQR payload generation with EMVCo-style TLV fields, server-owned
  VND amount, order-code transfer content, and CRC16-CCITT-FALSE checksum.
- Added mock webhook HMAC verification, idempotent repeated-webhook handling,
  customer payment ownership checks, and expired-payment handling.
- Added production safety controls so QR demo payment and the simulate endpoint
  are locked when `NODE_ENV=production`.
- Cleaned recent storefront UI issues caught by screenshots: public homepage
  internal counts, catalog result-count wrapping, ISBN overflow, header brand
  clipping, and footer support/copyright information.
- Added focused verification scripts for QR flow, production safety, secret
  scanning, and UI regression coverage.

## Environment Configuration

New development/sandbox variables are documented in `.env.example`:

```env
PAYMENT_MODE=demo
ENABLE_MOCK_PAYMENT=true
CASEFLOW_MERCHANT_NAME=CaseFlow Books

DEMO_BANK_BIN=970436
DEMO_BANK_NAME=Vietcombank
DEMO_BANK_ACCOUNT_NUMBER=0000000000
DEMO_BANK_ACCOUNT_NAME=
DEMO_PAYMENT_EXPIRES_MINUTES=15

MOCK_PAYMENT_WEBHOOK_SECRET=change-me-in-development
```

Production must not enable mock simulation. The runtime also denies QR demo
payments in production even if the demo variables are accidentally present.

## What Did Not Change

- No real bank, wallet, card, provider SDK, or payment-processing credential was
  integrated.
- No real bank account number or merchant account is stored in source code.
- No deep link opens a real banking or wallet application.
- Frontend still cannot decide the payable amount; the backend reads
  `orders.total_vnd`.
- Existing COD/bank-transfer/order-state simulation remains available where it
  was already part of checkout.
- The active catalog is still the 100-edition catalog; the requested
  500-edition repricing/expansion is a separate catalog-data migration scope.

## Residual Notes

- QR payment is intentionally a sandbox/demo feature and is locked from
  production settlement. A real provider requires a separate ADR, official
  provider SDK/webhook contract, reconciliation, refund handling, audit logs,
  rate limiting, and security review.
- The local QR verifier creates temporary customer/order/payment records and
  cleans them after the run. If a future verifier fails mid-run, release cleanup
  must be rerun before tagging.
- The 500-book expansion, realistic repricing, and FlowSync-inspired UI
  humanization request are queued as separate post-release scopes so the QR
  payment release does not mix payment schema risk with catalog-data risk.
