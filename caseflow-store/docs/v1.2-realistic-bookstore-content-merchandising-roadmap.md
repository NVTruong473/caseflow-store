# CaseFlow Books v1.2 Realistic Bookstore Content And Merchandising Roadmap

- Status: Accepted
- Date: 2026-07-17
- Baseline release: `v1.1.0` at commit `90a9907`
- Governing ADR:
  [ADR-0007](adr/0007-realistic-bookstore-content-merchandising-upgrade.md)
- Target release: `v1.2.0`

## Purpose

This roadmap turns the released 100-edition CaseFlow Books catalog into a more
credible, visually distinct, provenance-aware, and intentionally merchandised
bookstore. It improves the existing bookstore rather than adding another broad
feature layer.

This is a post-release versioned iteration. It must not be presented as Day
41+, another 20-day project, or additional retrospective journal history.

## Verified Baseline

The roadmap starts from facts observed in the `v1.1.0` release:

- the production application is deployed at
  `https://caseflow-store.vercel.app`;
- the catalog contains 100 sellable editions and supports real book works,
  English/Vietnamese edition relationships, filters, sorting, detail pages,
  cart entry, checkout, and admin catalog operations;
- every edition currently resolves to one internal placeholder cover;
- some edition facts are absent or intentionally generic, and visible product
  pages can show values such as `TBC` or a demo publisher;
- homepage, catalog, and detail layouts are responsive and tested, but repeated
  covers make products difficult to recognize and compare;
- the database already supports source notes, cover source types, localized
  alt text, work/edition separation, `is_featured`, and stable cover references;
- `v1.1.0` passed TypeScript, lint, production build, 20 local Playwright tests,
  a five-test production subset, accessibility/mobile checks, and production
  smoke tests.

## Success Definition

`v1.2.0` is successful when the site feels like a trustworthy specialist
bookstore while remaining honest about what it does not operate.

Required release outcomes:

- exactly 100 active sellable editions remain available unless an approved
  task records a justified replacement;
- all active works and editions have a reviewed provenance record;
- all active editions have bilingual project-written summaries and meaningful
  localized cover alt text;
- zero active catalog cards or detail pages use the generic placeholder as the
  primary image;
- every displayed edition-specific fact is traceable to one edition source;
- no visible `TBC`, seed/debug label, copied retailer blurb, fake rating, fake
  sold count, unsupported bestseller claim, or invented ISBN remains;
- homepage shelves and labels are editorially selected or derived from current
  catalog/order data under documented rules;
- operators can find content-quality gaps and manage the approved
  merchandising state without database-console work;
- the released checkout, auth, order, staff/admin, and API-security behavior
  has no regression;
- representative mobile and desktop visual checks show stable, legible,
  non-overlapping layouts with distinct product identity;
- production verification, documentation, rollback evidence, and the
  `v1.2.0` tag are complete.

## Non-Negotiable Guardrails

- Work one `V12-*` task at a time and verify it before starting the next task.
- Runtime implementation starts only after `V12-T02` passes and the user
  confirms `V12-T03`.
- Keep the v1.1 architecture and API/security boundaries unless a new ADR is
  accepted for a verified blocker.
- Do not change or retag `v1.1.0`.
- Keep VND authoritative and preserve server-side price, stock, promotion,
  shipping, tax, fee, and total calculation.
- Do not trust browser-supplied prices, totals, stock, role, or order status.
- Do not copy commercial covers, retailer/publisher descriptions, reviews, or
  protected excerpts without explicit recorded permission.
- Do not treat an API image URL or search result as proof of reusable rights.
- Do not fabricate ISBNs, publisher relationships, ratings, sold counts,
  bestseller rankings, market prices, or delivery promises.
- Do not add real payment credentials, e-book downloads, DRM, customer reviews,
  wishlist, AI recommendations, marketplace behavior, or a general-purpose CMS
  under this roadmap.
- Keep the generic placeholder available only as a runtime error fallback.
- Do not delete the v1.1 seed or previous assets before v1.2 production smoke
  and rollback checks pass.

## Quality And Evidence Rules

Every task must:

- record changed files, commands, results, risks, and the next task in both
  `.agent` mirrors;
- keep root/app documentation mirrors byte-identical where mirrors exist;
- run `git diff --check`;
- run `npm run lint` for code, data, schema, or script changes;
- run `npx tsc --noEmit` for TypeScript contract, data, repository, or UI
  changes;
- run `npm run build` for shared runtime, route, image, or integration changes;
- run focused Playwright or an equivalent deterministic browser script for the
  affected user/admin flow;
- save machine-readable reports and representative screenshots under a task
  artifact directory when the task changes visual or production behavior.

## Rollback Strategy

- Git: keep `v1.1.0` immutable and retain its deployment as the known-good code
  baseline.
- Database: export pre-v1.2 catalog/merchandising tables and record counts,
  constraints, policies, and checksums before writes.
- IDs: preserve work/edition IDs used by historical order snapshots and public
  URLs; use mappings or redirects for any approved slug replacement.
- Assets: keep v1.1 placeholder and previous paths until v1.2 production smoke
  passes; assets are additive until release acceptance.
- Migration: prefer additive schema and idempotent upserts; document the exact
  down/disable path before applying production SQL.
- Operations: never roll back by deleting customer, order, promotion,
  inventory-adjustment, or auth data.
- Deployment: if production smoke fails, restore the last known-good Vercel
  deployment and execute only the reviewed catalog/schema rollback steps.
