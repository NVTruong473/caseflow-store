# v1.13.1 - Order Response Reliability Patch

Date: 2026-07-24

## Summary

This patch corrects pending-order language, removes avoidable order-response
latency, and makes public catalog reads independent from customer auth cookies.
It also records a complete Production staff/admin simulated-transfer UAT.

## Customer Experience

- New pending orders now say `Order received` /
  `Don hang da duoc ghi nhan`; the interface no longer implies that staff has
  already confirmed the order.
- The account inbox is named `Account notifications` /
  `Thong bao tai khoan`, separating lifecycle messages from profile editing.
- Order history and notifications continue to show distinct order and payment
  states without requiring the customer to remember an order code.

## Reliability And Architecture

- External email/SMS outbox delivery is scheduled with Next.js `after()` after
  the committed order response. Transactional in-app events remain available
  immediately, while disabled external channels no longer delay checkout.
- Public catalog and merchandising reads use a server-side anonymous Supabase
  data client. Customer auth cookies are no longer copied into unrelated
  parallel public queries.
- Production UAT navigation and login use bounded retries for external-network
  instability. Order creation retries retain the same `checkoutAttemptId`, so
  the server's idempotency contract prevents duplicate orders.
- SEO verification now checks the current bilingual bookstore metadata and can
  write release-scoped evidence without changing the production contract.

## Operations Acceptance

- A temporary customer created two server-priced simulated bank-transfer
  orders.
- Staff confirmed one order and remained blocked from admin-only settings.
- Admin rejected the other order with a persisted risk-review reason.
- Repeated same decisions were idempotent; invalid reverse decisions returned
  `409`.
- Customer history and eight account notifications reflected both outcomes.
- Cleanup removed two orders and three temporary users and restored both
  affected stock records.

## Security And Production Controls

- Customer transfer-decision access returned `403`; staff settings returned
  `403`; admin settings returned `200`.
- All eight unauthenticated notification/operations boundaries returned the
  stable `401/UNAUTHORIZED` contract.
- QR simulation remains unavailable in Production; the runtime boundary
  returned `401`.
- Secret scan checked 1,473 files with zero findings.
- Public asset metadata scan checked 565 files with zero findings.
- No bank account, real payment provider, SMTP/SMS credential, or external
  delivery claim was added.
