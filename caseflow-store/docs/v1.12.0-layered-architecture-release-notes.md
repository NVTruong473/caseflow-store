# v1.12.0 - Layered Architecture And E2E Regression Hardening

Release date: 2026-07-22

## Summary

`v1.12.0` improves the architecture defensibility of CaseFlow Books without
changing the customer-facing commerce flow. The order creation API now follows
a clear Controller -> Use Case -> Repository boundary, and the repository gains
an automated architecture verifier to keep that boundary from drifting.

This release is intentionally not a classic MVC rewrite. CaseFlow Books remains
a Next.js modular monolith because that matches the deployed product and the
current Vercel/Supabase runtime.

## What Changed

- Added ADR-0014 for the layered architecture decision and guardrails.
- Documented the route-layering inventory and conventions for controller,
  use-case, DTO/API, repository, and validation responsibilities.
- Extracted `POST /api/orders` orchestration into
  `createBookOrderUseCase`.
- Kept the API response envelope and stable error codes unchanged.
- Added `apiFromUseCaseResult` to map typed use-case failures back to the
  existing `{ data, error, meta }` response shape.
- Added `npm run verify:architecture` to block use-case/repository imports from
  UI, feature, app-route, and Next route APIs.
- Re-ran checkout, API, payment, security, asset, and full Playwright
  regression gates after the refactor.

## Runtime Behavior

- No database migration.
- No customer checkout behavior change.
- No payment behavior change.
- No auth or role-boundary change.
- No public API envelope change.
- No production demo-payment enablement.

## Remaining Limitations

- The app is still a modular monolith, not a separate enterprise backend.
- Only the highest-risk order creation route was extracted in this release;
  several admin/customer routes still use direct controller-to-repository
  calls where the risk is lower.
- Production email/Supabase custom SMTP remains blocked until real SMTP and
  Supabase Management API credentials are supplied.
