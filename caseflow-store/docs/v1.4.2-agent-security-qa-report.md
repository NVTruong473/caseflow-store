## Scope

This patch uses `browser-use/web-ui` and `agentlabs-dev/auto-inspector` as
external QA workflow references. It does not add either project as a runtime
dependency, Docker sidecar, hosted service, or LLM-powered production feature.

That boundary matters because both tools are designed to drive browsers and can
touch authenticated sessions, form inputs, screenshots, logs, and secrets if
used carelessly. For CaseFlow Books, their useful contribution is QA process
design, not direct production integration.

## Report 1 - browser-use/web-ui Applicability

Source reviewed: `https://github.com/browser-use/web-ui`.

Useful ideas adopted:

- Browser-agent exploratory testing should be run from a disposable test
  browser context, not a real user Chrome profile.
- Long-running browser sessions are useful for audit trails, but they must not
  carry real customer/admin credentials.
- Visual evidence matters: screenshots and step records should be saved next
  to the verifier report.

Decision:

- Do not integrate `browser-use/web-ui` into the app or CI right now.
- Keep it as a manual/exploratory QA option for future local-only audits.
- Never point it at real customer accounts, real admin accounts, saved personal
  browser profiles, or production secrets.

Mapped CaseFlow action:

- Existing Playwright-based final QA remains the authoritative automated
  browser audit.
- `SECQA-T01` reused this principle by running final QA smoke after the security
  header patch and preserving artifacts under `.agent/artifacts/secqa-t01/`.

## Report 2 - auto-inspector Applicability

Source reviewed: `https://github.com/agentlabs-dev/auto-inspector`.

Useful ideas adopted:

- QA scenarios should be phrased as user stories with expected outcomes.
- Scenario runs should produce a clear report, not just a pass/fail terminal
  line.
- Secrets and credentials must be handled as test variables and not displayed
  in logs or sent to an agent when not necessary.

Decision:

- Do not self-host Auto Inspector for CaseFlow yet. Its README states the
  project is still in development and not ready to self-host.
- Do not send CaseFlow production admin/customer credentials into an autonomous
  third-party agent runner.
- Use its story-driven approach to define repeatable CaseFlow smoke scenarios.

Mapped CaseFlow action:

- `scripts/verify-final-post-release-qa.ts` already covers user-story style
  flows: public routes, catalog discovery, product detail, cart/checkout
  boundary, account/admin boundary, assistant guidance, tracking states, state
  previews, content safety, and overflow.
- `scripts/verify-security-posture.ts` adds a security-specific scenario
  layer: public page headers, protected page cache policy, API cache policy,
  admin API unauthorized boundary, CSP, HSTS, no framework fingerprint header,
  and permissions policy.

## Report 3 - Security Policies Applied

Runtime security hardening added in `next.config.ts`:

- `Content-Security-Policy` with `default-src 'self'`, `object-src 'none'`,
  `base-uri 'self'`, `frame-ancestors 'none'`, same-origin form actions, local
  image/font allowances, and Supabase-only remote connect allowances.
- `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`.
- `X-Frame-Options: DENY`.
- `X-Content-Type-Options: nosniff`.
- `Referrer-Policy: strict-origin-when-cross-origin`.
- `Permissions-Policy` disabling camera, microphone, geolocation, payment, USB,
  motion sensors, and browsing topics.
- `Cross-Origin-Opener-Policy: same-origin`.
- `Cross-Origin-Resource-Policy: same-origin`.
- `X-Permitted-Cross-Domain-Policies: none`.
- `X-XSS-Protection: 0`.
- `X-Powered-By` disabled through Next.js config.
- `Cache-Control: no-store, max-age=0` for API, account, admin, checkout, and
  order tracking surfaces.

Security policy posture:

- Server-side authorization remains the real boundary for customer/admin/staff
  access.
- Supabase service-role usage remains server-only.
- Browser cart values remain untrusted and are still revalidated by the server.
- Payment methods remain simulated; the app still does not collect card, bank,
  wallet, OTP, or payment-provider credentials.
- No schema migration, new dependency, external agent integration, or
  production data mutation was introduced.

Known residual risks:

- CSP still allows inline scripts/styles because tightening CSP with nonces in
  a Next.js App Router app requires a separate compatibility pass. The current
  policy still blocks object embeds, framing, foreign form posts, broad remote
  connections, and many browser capability classes.
- There is still no WAF/rate limit/CAPTCHA layer for public order tracking,
  login attempts, or checkout submission.
- There is no MFA enforcement for admin/staff accounts.
- There is no append-only operational audit log for admin/staff mutations.
- No real breach notification, data retention automation, or abuse monitoring
  workflow exists.

## Questions To Reach 95% Confidence For The Next Security Phase

1. Will this site keep using simulated payments only, or do you plan to connect
   a real payment provider within the next phase?
2. Do admin/staff accounts need MFA before you share the demo with recruiters
   or outside testers?
3. Should public order tracking remain available by order code plus contact, or
   should it move to signed links/account-only lookup?
4. Do you want a strict CSP nonce implementation even if it requires a deeper
   Next.js middleware/rendering compatibility pass?
5. Are you willing to add a real logging/monitoring service for security
   events, or should all evidence remain repository-local for now?
6. What is the maximum acceptable friction for users: CAPTCHA/rate limits on
   tracking and auth, or smoother UX with higher abuse risk?

## Top 0.1% Improvement Plan

1. Threat model the real business boundary: customer PII, admin access, order
   operations, promotions, inventory, checkout, and public tracking.
2. Add rate limiting on auth, tracking, checkout, customer profile, and admin
   mutation endpoints.
3. Add admin/staff MFA and session-risk policy before giving operations access
   to anyone outside the developer machine.
4. Implement nonce-based strict CSP and remove broad inline/eval allowances.
5. Add append-only audit events for admin/staff mutations and customer
   cancellation.
6. Add error monitoring and alert routing for auth failures, unexpected 5xx,
   order mutation anomalies, and RLS/permission denials.
7. Add backup/restore verification and data-retention policy for customer
   profile/order data.
8. Add CI security gates: dependency audit, secret scan, header verifier,
   Playwright smoke, and release cleanup.
9. Run quarterly manual security review using OWASP ASVS as the checklist.
10. Only after these controls exist, consider autonomous agent QA as an
    isolated, disposable test environment workflow.

## Challenging The Original Frame

The problem is not "add AI tester repos and security policies so hackers cannot
attack." That framing is too broad and can create false confidence.

The sharper problem is: CaseFlow is now public enough that every new testing
tool, browser agent, and security feature can either reduce risk or introduce a
new one. A browser agent with real sessions is itself a privileged actor. A CSP
that is copied blindly can break checkout. A security checklist that is not
verified by headers, route behavior, and production smoke is just paperwork.

So the correct bar is evidence: which routes changed, which headers exist, what
flow still works, what data remains protected, which risks remain open, and
what would need a deeper ADR before the next hardening step.
