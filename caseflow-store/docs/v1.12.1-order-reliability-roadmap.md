# v1.12.1 - Atomic Order Reliability Patch Roadmap

Date: 2026-07-22

## Objective

Close the post-`v1.12.0` transaction gap in order creation so a checkout retry
is idempotent and signup voucher redemption commits atomically with the order.

## Non-Goals

- No storefront redesign or new customer feature.
- No price, promotion value, payment, shipping, auth, or role-policy change.
- No external provider integration.
- No destructive database migration.
- No public API envelope change.

## Tasks

### ORDER-RELIABILITY-T01 - Accept Atomic Order ADR And Roadmap

Acceptance criteria:

- The current split-transaction failure mode is documented.
- The decision defines database-enforced idempotency and atomic voucher use.
- Rollout and compatibility guardrails are explicit.

Verification:

- ADR is indexed and roadmap exists.

### ORDER-RELIABILITY-T02 - Add Checkout Attempt Contract And Atomic RPC

Acceptance criteria:

- Checkout requests carry a validated UUID attempt identifier.
- An additive migration stores attempt IDs and provides a versioned RPC.
- The RPC returns an existing order for a repeated customer/attempt pair.
- Order, items, and optional voucher redemption share one transaction.

Verification:

- Migration verifier covers structure, grants, uniqueness, atomic voucher
  update, and rollback plan.
- TypeScript database contracts match the RPC.

### ORDER-RELIABILITY-T03 - Integrate Controller, Use Case, Repository, And UI

Acceptance criteria:

- The browser reuses one attempt ID across retries and clears it on success.
- The controller remains thin and the API envelope is unchanged.
- The use case passes only server-validated customer, totals, cart, and
  promotion data to the repository.
- Pre-order voucher reservation and post-order confirmation writes are removed
  from order creation.

Verification:

- Architecture verifier, lint, TypeScript, and focused checkout tests pass.

### ORDER-RELIABILITY-T04 - Add Retry And Atomicity Regression Coverage

Acceptance criteria:

- Same attempt submitted twice returns the same order.
- Only one order row exists for the attempt.
- A signup voucher is linked to that order exactly once.
- Invalid, expired, mismatched, and already-used account vouchers cannot create
  a discounted order.

Verification:

- Dedicated database/API verifier passes and cleans up its test data.
- Checkout and API error Playwright suites pass.

### ORDER-RELIABILITY-T05 - Apply Migration And Run Full Local Gate

Acceptance criteria:

- A pre-migration count/safety check is recorded.
- The additive migration applies successfully and is idempotently verifiable.
- Lint, TypeScript, build, full E2E, architecture, security, QR safety, asset,
  and dependency gates pass after the final code.

Verification:

- Evidence is stored under `.agent/artifacts/order-reliability-*`.

### ORDER-RELIABILITY-T06 - Deploy And Release v1.12.1

Acceptance criteria:

- Production deployment is ready and aliased.
- Production checkout retry verification passes without leaving duplicate test
  orders or voucher state.
- Production smoke, security posture, QR lock, and full Playwright pass.
- Annotated tag and GitHub Release describe the reliability patch truthfully.

Verification:

- Git, GitHub, Vercel, runtime evidence, and release metadata agree.
