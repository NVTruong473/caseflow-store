# v1.12 - Layered Architecture And E2E Regression Hardening Roadmap

Date: 2026-07-22

## Objective

Improve CaseFlow Books architecture defensibility by making high-risk mutating
APIs follow a clear Controller -> Use Case -> Policy/Validation -> Repository
boundary while preserving runtime behavior and release evidence.

## Non-Goals

- No classic MVC rewrite.
- No framework change.
- No schema migration unless a verified blocker appears.
- No feature expansion.
- No payment, shipping, auth, or role-behavior change.
- No public API envelope change.

## Tasks

### ARCH-LAYER-T01 - Adopt Layered Architecture ADR

Acceptance criteria:

- ADR explains why the project uses layered architecture, not textbook MVC.
- ADR preserves the Next.js modular monolith.
- ADR defines allowed layer responsibilities and guardrails.

Verification:

- ADR exists and is indexed.

### ARCH-LAYER-T02 - Audit API Layering Baseline

Acceptance criteria:

Verification:

- Inventory doc exists.

### ARCH-LAYER-T03 - Document Layering Conventions

Acceptance criteria:

### ARCH-LAYER-T04 - Extract Book Order Creation Use Case

Acceptance criteria:

- `POST /api/orders` becomes a thin controller.
- Use case owns auth, profile/contact checks, cart validation, promotion
  evaluation, signup voucher reservation/rollback, server-owned totals, and
  order creation.
- API response envelope and error codes do not change.

Verification:

- Lint, TypeScript, build, checkout E2E, API error E2E.
